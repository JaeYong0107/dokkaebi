# dokkaebi 동작 확인 시나리오

5개 TODO 항목(Prisma/Auth/API/결제/관리자 CRUD) 이 의도대로 동작하는지
브라우저에서 따라가며 확인하는 E2E 시나리오. 각 단계는 **행동**과
**기대 결과**를 같이 적어두었다.

## 사전 준비

```bash
docker ps                      # dokkaebi-postgres Up (healthy)
npm run dev                    # http://localhost:3000
```

**테스트 계정** (seed 자동 생성)

| 역할 | 이메일 | 비밀번호 |
|---|---|---|
| 관리자 | admin@dokkaebi.kr | admin1234 |
| 일반 고객 | user@dokkaebi.kr | user1234 |
| 사업자 (승인됨) | biz@dokkaebi.kr | biz12345 |

시나리오 중간에 브라우저 탭을 두 개 쓰면 편하다.
`Tab A` = 사용자, `Tab B` = 관리자.

---

## Phase 1 — 비로그인 탐색 (공개 페이지 + 라우트 가드)

### 1-1. 홈
- **행동**: `http://localhost:3000/` 열기
- **기대**: 상품 카드 목록 렌더, 에러 배너 없음
  (`/api/orders` 는 비로그인 200 빈 배열)

### 1-2. 상품 목록 · 상세
- **행동**: 상단 네비 → 상품 → 카테고리 칩 "과일" 클릭
- **기대**: 과일 상품만 필터링. 상품 카드 클릭 → 상세 페이지 200

### 1-3. 장바구니 가드
- **행동**: `/cart` 직접 접근
- **기대**: `/login?callbackUrl=%2Fcart` 로 307 리다이렉트

### 1-4. 관리자 가드
- **행동**: `/admin` 직접 접근
- **기대**: 동일하게 `/login?callbackUrl=%2Fadmin` 리다이렉트

## Phase 2 — 회원가입 + 로그인 (Auth.js)

### 2-1. 회원가입 (일반)
- **행동**: `/signup` → 회원유형 "일반 고객" → 이메일·비밀번호·이름 입력 → 약관 동의 → 가입
- **기대**: 가입 직후 자동 로그인, 홈으로 이동

### 2-2. 유효성 검증
- **행동**: 다시 `/signup` 에서 같은 이메일로 재시도
- **기대**: "이미 가입된 이메일입니다" 409 메시지

### 2-3. 사업자 회원 전환 입력
- **행동**: 새 이메일로 `/signup`, "사업자" 라디오 선택
- **기대**: 상호/사업자등록번호 필드가 동적 등장, 미입력 시 제출 차단

### 2-4. 로그인 오류
- **행동**: `/login` 에 admin@dokkaebi.kr / wrongpass 입력
- **기대**: "이메일 또는 비밀번호가 올바르지 않습니다" 에러

### 2-5. 로그인 성공
- **행동**: admin 계정으로 로그인
- **기대**: 홈으로 이동

---

## Phase 3 — 사업자 쇼핑 (세션 기반 가격 분기)

### 3-1. biz 로그인
- **행동**: `biz@dokkaebi.kr / biz12345` 로그인

### 3-2. 도매가 확인
- **행동**: `/products/prod-lettuce-001` 열기
- **기대**:
  - 정가 18,500원
  - 사업자 할인 22% 적용된 가격(약 14,430원) 표시
  - "사업자" 뱃지가 보임

### 3-3. 장바구니 담기
- **행동**: 수량 4로 "장바구니 담기"
- **기대**: 상단 장바구니 카운트 4 증가

### 3-4. 장바구니 요약
- **행동**: `/cart` 이동
- **기대**:
  - 품목 1개, 사업자 단가 기준 합계
  - 사업자는 배송비 0원
  - 최소 주문 금액 바(50,000원) 미달 → 진행 바 일부만 채워짐
  - "주문하기" 버튼 비활성화

### 3-5. 최소 주문 충족
- **행동**: 수량 +/- 로 수량 조정해 합계 50,000원 이상으로 만들기
- **기대**: 바 100% + 버튼 활성화

### 3-6. 결제
- **행동**: "주문하기" → 결제 페이지 → 결제 수단 선택 → 동의 두 개 체크 → 하단 결제 버튼
- **기대**:
  - 버튼 "결제 진행 중..." 표시
  - 성공 시 `/orders/complete?orderId=...` 로 이동
  - 서버 로그에 `POST /api/orders 201` 확인 가능

## Phase 4 — 주문 관리 (본인 주문)

### 4-1. 주문 목록
- **행동**: `/orders`
- **기대**: 방금 만든 주문 + 시드 주문(SHIPPING/DELIVERED) 포함 3건

### 4-2. 주문 상세 · 배송 추적
- **행동**: 시드 주문 "SHIPPING" 클릭
- **기대**: 배송사 "롯데택배", 운송장 번호 표시, 타임라인 렌더

### 4-3. 마이페이지
- **행동**: `/mypage`
- **기대**:
  - 최근 주문 섹션에 방금 주문 포함
  - 자주 사는 상품 2건 (주문 수량 많은 순)
  - 빠른 재주문 바에 최근 주문의 상품 1~3개

### 4-4. 재주문
- **행동**: `/reorder?orderId=<방금 주문 id>` (마이페이지에서 연결된 버튼)
- **기대**: 품목 자동 채움, 수량 편집 가능

## Phase 5 — 관리자 콘솔

### 5-1. admin 로그인 (Tab B)
- **행동**: 로그아웃 → `admin@dokkaebi.kr` 로 로그인 → `/admin`
- **기대**: 대시보드 렌더
  - 오늘의 매출(현재 DB 주문 합계)
  - 오늘 주문 수, 최근 주문 4건 표시

### 5-2. 주문 관리
- **행동**: 사이드바 → "주문 관리"
- **기대**: 전체 주문 목록. Phase 3 에서 biz 가 만든 주문 포함.
  상태 탭(결제완료/준비중/배송중/완료/취소) 필터 동작

### 5-3. 사용자 관리
- **행동**: "사용자 관리"
- **기대**:
  - 전체 6명(관리자/일반 2+/사업자 1) 목록
  - 상단에 "사업자 승인 대기" 카운트
  - biz 사용자 행 → "승인 해제" 버튼 → 클릭 시 "대기" 로 변경
  - 다시 "승인" 클릭 → "승인됨"

### 5-4. 권한 부여 차단
- **행동**: admin 행의 "권한 내리기" 버튼 클릭
- **기대**: 빨간색 에러 "본인 계정의 관리자 권한은 내릴 수 없습니다"

### 5-5. 일반 회원에 사업자 승인
- **행동**: user@dokkaebi.kr 행 (승인 컬럼 "—") 근처에 승인 버튼이 없음을 확인.
  curl 로 직접 시도 시 409 반환 (UI에서는 버튼 자체가 숨김)

### 5-6. 상품 관리 · 신규 등록
- **행동**: "상품 관리" → "신규 상품 등록" → 모달에서 입력
  - 상품 ID: `prod-qa-test-001`
  - 상품명: "QA 테스트 상품"
  - 카테고리: 채소 (veg)
  - 정가: 10000, 사업자 할인율: 10
  - 단위: `1kg / 봉`, 원산지: `테스트`
  - 이모지: 🧪, 배경: `from-slate-200 to-slate-400`
  - 재고: 100
  - **등록**
- **기대**: 목록 새로고침 → 상단에 방금 등록한 상품 노출, 판매중 배지

### 5-7. 상품 편집
- **행동**: 해당 상품 "편집" → 정가를 20000 으로 변경 → 저장
- **기대**: 목록의 정가 20,000원으로 갱신

### 5-8. 판매 중지 토글
- **행동**: "중지" 버튼 → 다른 탭에서 `/products` 새로고침
- **기대**:
  - 관리자 목록은 해당 행이 "중지" 상태
  - 공개 상품 페이지에서는 isActive=false 라 노출되지 않거나 비활성 스타일
- **되돌리기**: "판매 재개"

### 5-9. 정책 변경 (실험)
- **행동**: "정책 관리" → 사업자 최소 주문을 **80000** 으로 변경 → 저장
- **기대**: "정책값이 저장되었습니다" 배너
